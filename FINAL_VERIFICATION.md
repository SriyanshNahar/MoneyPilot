## Methodology (what "verification" actually means below)

1. **Static/code verification** — reading the actual current file, not memory of it.
2. **Build verification** — `flutter analyze`, `flutter test`, `flutter build apk --debug`, all re-run fresh in this pass (not reused from earlier in the conversation).
3. **Live backend verification** — direct HTTPS calls to the project's real Supabase REST API (`/rest/v1/...`) and Edge Functions (`/functions/v1/...`) using the app's own anon key, to check what's actually deployed vs. only committed to git.
4. **Live UI verification** — a real `flutter build web` + `flutter run -d web-server`, opened in an actual browser tab, clicked through by hand. Real screenshots below. Stopped after the incident above, so **screens that require being logged in were not re-tested live** — those are marked accordingly rather than guessed at.

No Android emulator was usable (the only configured AVD failed to boot to a usable state in this sandbox), so mobile-native behavior (biometrics, FCM push, deep links, local notifications) is analyze/build-verified only, not runtime-verified on-device in this pass.
